# 04 — 多元协同治理

## 4.1 模块概述

多元协同治理是 BiDA-KG 的**核心分析引擎**。基于四层知识图谱，进行风险主体识别 → 群体发现 → 风险传导路径分析 → 合规匹配 → 协同治理社区报告生成。整体架构为 **DRA-MA (Dynamic Reasoning & Multi-stage Analysis)** 管线，按阶段依次执行意图识别、实体解析、图谱检索、社区发现、风险分析、合规研判和报告生成。

系统提供**三个对外开放的 REST API**，既可供外部系统集成调用，也是内部 SSE 实时管线 (`POST /api/v1/chat/unified-stream`) 的后台计算单元。

---

## 4.2 治理管线阶段

协同治理管线按以下阶段依次执行：

```
用户输入 query
    │
    ▼
[意图识别] 意图分类 (graph_qa / risk_analysis) + 实体抽取
    │
    ▼
[实体解析] 5级级联匹配 (精确 → 别名 → 包含 → 模糊 → LLM回退)
    │
    ▼
[图谱检索] 子图查询 → 实体统计 → 群体发现 (WCC / Louvain / HGT-GKMeans)
    │
    ▼
[风险分析] 风险路径识别 + 异常发现 + 群体路径联动
    │
    ▼
[合规研判] 法规匹配 + 合规指标评分 (一/二/三级指标)
    │
    ▼
[风险评分] 6维评分 (关系复杂度/风险关系/社区密度/传导/合规/证据) + LLM调整
    │
    ▼
[治理方案] 治理动作 + 升级规则 + 监控清单
    │
    ▼
[报告生成] 执行摘要 + Markdown报告 + 建议措施
```

每个阶段由独立的治理模块执行，模块之间通过 `GovernanceContext` 共享状态，通过 SSE `EventPublisher` 推送阶段性结果。

---

## 4.3 对外开放 API

三个 API 均采用 POST JSON 请求/响应模式，返回统一结构 `{ code: 0, msg: "success", data: {...} }`。它们既可直接对外暴露供第三方调用，也是内部 SSE 管线 (`/api/v1/chat/unified-stream`) 复用的计算单元。

### 4.3.1 API 概览

| API | 端点 | 用途 |
|-----|------|------|
| 风险实体群体发现 | `POST /api/v1/governance/community-discovery` | 种子实体 → k-hop 子图 → 社区检测 → 两级可视化图谱 |
| 风险传导路径分析 | `POST /api/v1/risk-paths` | 种子实体 → 风险路径到公司/人员/事件/法规 → 打分可解释高亮路径 |
| 合规评分与治理报告 | `POST /api/v1/governance/reports` | 综合查询/子图/社区/路径/异常 → 合规匹配/指标评分/治理方案/完整报告 |

### 4.3.2 风险实体群体发现

**端点**: `POST /api/v1/governance/community-discovery`

将种子实体名称或 Neo4j 节点 ID 输入系统，自动完成 k-hop 自我网络扩展 → 最大连通分量 (LCC) 裁剪 → 群体检测 (WCC / Louvain / HGT-GKMeans 三级自适应回退) → 社区聚合图，适用于风险群体画像和两级缩放可视化。

**请求参数**:

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `seedNames` | string[] | 否 | `[]` | 种子实体名称列表，支持中文 |
| `seedIds` | string[] | 否 | `[]` | 种子实体 Neo4j elementId 列表 |
| `maxHop` | int | 否 | `3` | k-hop 扩展跳数 (1-5) |
| `method` | string | 否 | `"auto"` | 群体检测算法: `auto` / `louvain` / `wcc` / `hgt_gkmeans` |
| `communityMode` | string | 否 | `"expanded"` | 群体模式: `expanded` / `global_seed` |
| `minCommunitySize` | int | 否 | `2` | 最小群体规模 |
| `maxNodes` | int | 否 | `300` | 子图最大节点数 (10-5000) |
| `pathLimit` | int | 否 | `5000` | k-hop 遍历路径数上限 (50-10000) |
| `relationWhitelist` | string[] | 否 | `[]` | 关系类型白名单，空数组=全部关系 |
| `responseMode` | string | 否 | `"full"` | 响应模式: `full` / `summary` |
| `includeRawSubgraph` | bool | 否 | `true` | 是否返回原始子图 |
| `includeCommunityGraph` | bool | 否 | `true` | 是否返回社区聚合图 |
| `includeHgtEmbedding` | bool | 否 | `false` | 是否包含 HGT 嵌入向量 |

**响应字段**:

| 字段 | 类型 | 说明 |
|------|------|------|
| `success` | bool | 是否成功 |
| `traceId` | string | 链路追踪 ID |
| `elapsedMs` | int | 耗时 (毫秒) |
| `selectedMethod` | string | 实际使用的检测算法 |
| `fallbackReason` | string \| null | 算法回退原因 |
| `seedNodes` | object[] | 被成功解析的种子节点 |
| `summary` | object | 摘要: `nodeCount`, `edgeCount`, `communityCount`, `seedCommunityId` |
| `warnings` | string[] | 警告信息列表 |
| `communities` | object[] | 所有检测到的群体 |
| `entityCommunityMap` | object | `{node_id: {communityId, role, isSeed, riskLevel}}` — 节点→群体映射 |
| `visualization` | object | 可视化建议: `defaultView`, `suggestedLayout`, `highlightCommunityId` |
| `connectedSubgraph` | object | 最大连通分量 (nodes + edges) |
| `subgraph` | object \| null | 原始 k-hop 子图 |
| `communityEdges` | object[] | 社区间桥接边 |
| `communityGraph` | object | 社区聚合图 (nodes + edges)，用于前端两级缩放 |

**群体检测算法说明**:

- `auto`: 由系统自适应选择 —— 节点数 < 30 使用 WCC，≥ 30 使用 Louvain，失败时回退至 WCC
- `louvain`: networkx Louvain 群体检测
- `wcc`: 并查集弱连通分量 (Union-Find)
- `hgt_gkmeans`: 预计算 HGT 嵌入 + sklearn KMeans 聚类

**entityCommunityMap 格式 (强制)**:

```json
{
  "<node_id>": {
    "id": "<node_id>",
    "name": "实体名称",
    "type": "COMPANY",
    "communityId": 0,
    "role": "core",
    "isSeed": true,
    "riskLevel": "high"
  }
}
```

以节点 ID 为 key (非实体名称)，确保后续 `community_path` 映射的准确性。

### 4.3.3 风险传导路径分析

**端点**: `POST /api/v1/risk-paths`

输入用户查询或种子实体，系统通过治理管线自动完成实体解析 → 图谱检索 → 群体发现 → 风险路径识别和解释，返回候选路径、解释路径、合并路径、异常发现、群体信息和高亮图元。

**请求参数**:

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `query` | string | 否 | `""` | 用户自然语言查询或风险分析提示 |
| `focusEntities` | string[] | 否 | `[]` | 关注实体名称列表 |
| `sourceEntity` | string \| null | 否 | `null` | 指定来源实体 |
| `targetEntities` | string[] | 否 | `[]` | 指定目标实体列表 |
| `maxHop` | int | 否 | `3` | 最大分析跳数 (1-5) |
| `mode` | string | 否 | `"both"` | 分析模式: `node` / `community` / `both` |
| `relationTypes` | string[] | 否 | `[]` | 关系类型筛选 |
| `subgraph` | object \| null | 否 | `null` | 外部传入的已知子图 (复用已检索结果) |
| `communities` | object \| null | 否 | `null` | 外部传入的已知群体结果 |
| `entityCommunityMap` | object \| null | 否 | `null` | 外部传入的已知节点→群体映射 |
| `fileContent` | string \| null | 否 | `null` | 上传文件文本内容 (用于实体抽取) |
| `sessionId` | string | 否 | `""` | 会话 ID (复用已有上下文) |
| `roundId` | int | 否 | `1` | 对话轮次 |

**响应字段**:

| 字段 | 类型 | 说明 |
|------|------|------|
| `query` | string | 实际执行的查询文本 |
| `mode` | string | 分析模式 |
| `candidate_paths` | object[] | 候选风险路径 (图结构匹配) |
| `risk_paths` | object[] | 经 LLM 解释的风险路径 (含 `path_description`, `risk_level`, `confidence`) |
| `merged_paths` | object[] | 合并去重后的风险路径 (含 `community_path` 群体路径链) |
| `node_paths` | object[] | 节点级路径 (同 merged_paths) |
| `community_paths` | object[] | 社区级路径 |
| `anomaly_findings` | object[] | 异常发现列表 |
| `community_info` | object | 群体发现结果 |
| `entity_community_map` | object | 节点→群体映射 |
| `subgraph` | object | 证据子图 |
| `graph_highlight` | object | 高亮图元: `{node_ids: [...], edge_ids: [...]}` |

**风险路径对象结构**:

```json
{
  "path_id": "RP-001",
  "risk_level": "high",
  "affected_entities": ["实体A", "实体B"],
  "node_ids": ["n1", "n2", "n3"],
  "edge_ids": ["e1", "e2"],
  "community_path": [0, 1],
  "path_description": "实体A通过控股关系传导风险至实体B",
  "confidence": 0.85
}
```

`community_path` 字段由管线自动生成——将路径上的每个节点映射到其所属群体 ID (去重)，形成群体级风险传导链。该链可直接用于前端群体图谱高亮和群体标签联动展示。

### 4.3.4 合规评分与治理报告

**端点**: `POST /api/v1/governance/reports`

输入用户查询或已完成的中间分析结果 (群体、路径等)，系统运行完整治理管线，返回包含合规匹配、指标评分、风险分数、治理方案和执行摘要的结构化报告。

**请求参数**:

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `query` | string | **是** | — | 用户问题或治理报告主题 |
| `focusEntities` | string[] | 否 | `[]` | 关注实体列表 |
| `maxHop` | int | 否 | `3` | 最大分析跳数 (1-5) |
| `fileContent` | string \| null | 否 | `null` | 上传文件文本内容 |
| `communityResult` | object \| null | 否 | `null` | 外部传入的群体发现结果 (复用已有分析) |
| `riskPaths` | object \| list \| null | 否 | `null` | 外部传入的风险路径 (复用已有分析) |
| `complianceContext` | object \| null | 否 | `null` | 外部传入的合规上下文 |
| `exportFormats` | string[] | 否 | `[]` | 导出格式: `markdown` / `docx` / `pdf` |
| `sessionId` | string | 否 | `""` | 会话 ID |
| `roundId` | int | 否 | `1` | 对话轮次 |

**响应字段**:

| 字段 | 类型 | 说明 |
|------|------|------|
| `report_id` | string | 唯一报告编号 (如 `WIND-RPT-A1B2C3D4`) |
| `query_summary` | string | 查询摘要 |
| `executive_summary` | string | 社区治理执行摘要 |
| `community_info` | object | 群体发现结果 |
| `entity_community_map` | object | 节点→群体映射 |
| `risk_paths` | object[] | 风险传导路径列表 |
| `anomaly_findings` | object[] | 异常发现 |
| `compliance_matches` | object[] | 法规匹配结果 (含法规名称/条款/违规认定/建议措施) |
| `risk_scores` | object | 6维风险评分 + LLM调整 |
| `governance_plan` | object | 治理方案: `actions` + `escalation_rules` + `monitoring_checklist` |
| `recommendations` | object[] | 建议措施列表 |
| `markdown_report` | string | Markdown 格式完整报告 |
| `subgraph_summary` | object | 证据子图摘要: `{node_count, edge_count}` |
| `export_files` | object | 导出文件: `{markdown: {filename, content}, ...}` |
| `source_events` | object | 数据来源标记: `{has_community, has_risk_paths, has_governance}` |

**风险评分维度**:

| 维度 | 说明 |
|------|------|
| 关系复杂度 | 子图节点/边密度与关系类型多样性 |
| 风险关系 | 高风险关系 (被执行/失信/担保) 占比 |
| 社区密度 | 社区内部紧密度与跨社区桥接度 |
| 传导风险 | 风险路径长度、数量和置信度加权 |
| 合规风险 | 法规匹配数量与违规严重程度 |
| 证据充分性 | 证据链完整度与验证得分 |

最终评分由上述 6 维加权计算后，结合 LLM 全局研判调整得出。

**治理方案结构**:

```json
{
  "actions": [
    {
      "target": "实体A",
      "risk_issue": "股权代持风险",
      "measure": "核查实际控制人与一致行动人关系",
      "priority": "urgent",
      "department": "合规部"
    }
  ],
  "escalation_rules": [
    {
      "condition": "关联方交易金额 > 1000万",
      "action": "升级至董事会风险委员会",
      "timeline": "24小时内"
    }
  ],
  "monitoring_checklist": [
    "每日监控重点关注实体关联变动",
    "每周更新风险传导路径状态"
  ]
}
```

**API 间复用关系**:

三个 API 可独立调用，也可组合复用:
- `community-discovery` → 获取 `entityCommunityMap` 和 `subgraph` → 传入 `risk-paths` 的 `subgraph`/`communities`/`entityCommunityMap` 参数，跳过图谱检索阶段
- `risk-paths` → 获取 `risk_paths` 和 `anomaly_findings` → 传入 `reports` 的 `riskPaths` 参数，跳过风险分析阶段
- 这种设计使得 API 既能独立对外服务，也能在内部 SSE 管线中作为阶段性计算单元复用

## 4.4 协同治理统一管线 (UnifiedEngine)

协同治理模块的实际运行入口有两个层面:

| 层面 | 入口 | 说明 |
|------|------|------|
| **REST API** | 三个独立端点 (4.3 节) | 外部集成、同步调用、可复用 |
| **SSE 流式管线** | `POST /api/v1/chat/unified-stream` | 实时推送、交互式前端、16种事件类型 |

REST API 底层复用 SSE 管线的计算模块 (`_collect_unified_analysis`)，两者共享 `GovernanceContext` 状态结构和治理模块链。

### 4.4.1 治理模块链

每个管线阶段由独立的治理模块实现，通过 `GovernanceContext` 数据类传递共享状态:

| 模块 | 输出事件 | 依赖 |
|------|---------|------|
| IntentModule | `intent`, `entities` | — |
| EntityModule | `entity_resolved` | intent |
| CommunityModule | `subgraph`, `entity_stats`, `community`, `entity_community_map`, `expanded_community` | entity |
| RiskPathModule | `risk_paths`, `anomaly_findings` | community |
| ComplianceModule | `compliance`, `compliance_scores`, `compliance_indicators` | risk_path |
| ScoringModule | `scoring` | compliance |
| GovernanceDecisionModule | `governance` | scoring |
| ReporterModule | `report` | governance |
| EvidenceModule | `evidence_chains` | risk_path |

### 4.4.2 SSE 事件类型 (16种)

| 事件 | 阶段 | 说明 |
|------|------|------|
| `stage` | 全局 | 当前阶段变更 |
| `entities` | 实体解析 | 已对齐实体 + 未解析实体 |
| `subgraph` | 图谱检索 | 证据子图 (nodes + edges) |
| `entity_stats` | 图谱分析 | 实体统计信息 |
| `community` | 群体发现 | 社区列表、成员数、算法 |
| `entity_community_map` | 群体发现 | 实体→社区归属 + 角色 (bridge/hub/member) |
| `expanded_community` | 群体发现 Phase B | 扩展社区检测完整结果 (communityGraph + communityEdges) |
| `candidate_risk_paths` | 风险分析 | 候选风险路径 |
| `risk_paths` | 风险分析 | 确认的风险路径 + 描述 + 置信度 + community_path |
| `anomaly_findings` | 风险分析 | 异常发现列表 |
| `compliance` | 合规分析 | 法规匹配结果 |
| `compliance_scores` | 合规分析 | 合规指标得分 |
| `compliance_indicators` | 合规分析 | 一/二/三级合规指标详情 |
| `scoring` | 风险评分 | 6维评分 + LLM调整 |
| `governance` | 治理方案 | 治理动作 + 升级规则 + 监控清单 |
| `report` | 报告生成 | executive_summary + markdown_report + recommendations |
| `done` / `error` | 终态 | 完成或错误 |

### 4.4.3 SSE 事件信封格式

```json
{
  "event_id": "evt-xxx",
  "session_id": "sess-xxx",
  "round_id": 1,
  "stage": "risk_analysis",
  "type": "risk_paths",
  "status": "success",
  "data": { ... },
  "error": null,
  "timestamp": "2026-06-06T10:00:00Z"
}
```

### 4.4.4 群体发现双阶段机制

群体发现分为两个阶段:

- **Phase A**: 对已解析实体进行实体统计 + 子图排除后的剩余实体进行群体检测 + PageRank 中心性计算
- **Phase B**: 基于完整的 k-hop 自我网络扩展 + 群体检测 (WCC/Louvain/HGT-GKMeans) + 社区聚合图构建，并通过 `expanded_community` SSE 事件推送完整结果供前端两级缩放使用

## 4.5 LLM 集成规范

### 4.5.1 模型配置

```env
LLM_API_KEY=sk-xxx
LLM_BASE_URL=https://api.deepseek.com/v1
LLM_MODEL=deepseek-v4-flash
```

### 4.5.2 调用策略

| 场景 | Temperature | 原因 |
|------|-------------|------|
| Text-to-Cypher | 0.0 | 需要确定性输出 |
| 风险分析 | 0.3 | 需要一定创造性但保持可控 |
| 报告撰写 | 0.5 | 需要流畅自然的语言表达 |
| 实体抽取 (NER) | 0.0 | 需要精确的结构化输出 |

### 4.5.3 私有化部署要求

- 生产环境中 LLM 应私有化部署，确保敏感数据不出域
- 调用外部 API 时，输入数据需经过去敏处理 (企业名称可用代号替代，个人姓名/证件号脱敏)
- 所有 LLM 调用需记录日志，包括请求时间、Prompt 摘要、Token 消耗、响应耗时

## 4.6 异常发现

### 4.6.1 主动异常检测

系统定期 (每日/每小时) 对全图或特定子图运行异常检测算法:

| 检测类型 | 算法 | 说明 |
|---------|------|------|
| **资金环流检测** | 有向环检测 (DFS/Tarjan) | 识别 A→B→C→A 的循环资金流 |
| **社区欺诈检测** | Louvain 社区发现 + 密度分析 | 发现紧密耦合的异常关联群 |
| **突发实体检测** | 时序分析: 节点度突变 | 某实体短期内关联关系剧增 |
| **结构洞检测** | Betweenness Centrality | 识别网络中关键"桥梁"节点 |
| **异常路径检测** | LLM-based 模式匹配 | 用自然语言描述异常模式，在图中搜索匹配 |

### 4.6.2 触发式分析

当系统监测到以下事件时，自动触发协同分析:

- 图谱中新增"被执行"、"失信"、"破产"类事件节点
- 某重点监控实体发生属性重大变更 (法人/股权/经营范围)
- 负面舆情事件达预设热度阈值
- 监管机构发布新的风险提示或法规

## 4.7 协同治理社区报告

### 4.7.1 报告生成流程

```
意图识别 → 实体解析 → 群体发现 → 风险传导路径 → 合规分析 → 风险评分 → 治理方案 → 完整报告
```

### 4.7.2 报告字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `report_id` | string | 唯一报告编号 |
| `executive_summary` | string | 社区治理结论摘要 |
| `communities` | object[] | 群体发现结果 |
| `risk_paths` | object[] | 风险传导路径 |
| `compliance_matches` | object[] | 合规匹配结果 |
| `risk_scores` | object | 6维风险评分 |
| `governance_plan` | object | 治理建议、优先级和升级规则 |
| `recommendations` | object[] | 建议措施 |
| `markdown_report` | string | Markdown 格式完整报告 |

### 4.7.3 治理建议类型

| 建议类型 | 说明 |
|---------|------|
| **风险核查** | 对关键主体、资金链路和股权关系进行重点核查 |
| **合规补充** | 补充披露、关联关系说明或内部决策材料 |
| **持续观察** | 跟踪风险传导路径上的主体状态变化 |
| **升级研判** | 对高风险路径触发进一步研判和人工复核 |

### 4.7.4 报告导出

- **在线查看**: 浏览器中渲染的 HTML 报告 (ProLayout + 组件化)
- **导出格式**: PDF (Puppeteer 服务端渲染) / Word (docx 模板) / Markdown

## 4.8 风险可视化看板

基于 AntV G6 v4.8 的知识图谱可视化，支持分层展示、交互式探查和社区两级缩放。

### 4.8.1 核心可视化

**社区两级缩放 (GlobalCommunityGraph)**:

- **Level 1 — 社区聚合图**: 渲染 `communityGraph.nodes` + `communityGraph.edges`，节点大小按成员数缩放，颜色按社区 ID 着色，边权重反映社区间桥接强度
- **Level 2 — 成员展开图**: 点击社区节点 → 展开显示该社区的成员实体节点，按角色 (core/bridge/member) 和种子状态差异化样式，桥接边用虚线标识
- **风险路径联动**: 点击风险路径 → `community_path` 高亮 + 展开状态下的实体节点高亮 + 社区 Tag 链展示

**节点类型着色**:

| 节点类型 | 颜色 | 说明 |
|---------|------|------|
| COMPANY | `#1890ff` | 公司主体 |
| PERSON | `#722ed1` | 人员 |
| EVENT | `#f5222d` | 事件 |
| SUB_EVENT | `#fa8c16` | 子事件 |
| RiskFeature | `#eb2f96` | 风险特征 |
| Law / Regulation | `#13c2c2` | 法规 |

### 4.8.2 交互操作

| 操作 | 行为 | 技术要求 |
|------|------|---------|
| **节点点击** | 展开节点详情面板 (属性列表 + 1跳邻居) | < 200ms 响应 |
| **节点下钻** | 以当前节点为中心，展开 N 跳邻居 | 默认 1 跳，可配置 2-3 跳 |
| **关系高亮** | 鼠标悬停关系边，高亮显示关联节点和路径 | 高亮 + 其他节点半透明 |
| **路径追溯** | 选中两个节点，自动查询并高亮所有路径 | 调用 `/kg/path` API |
| **框选/圈选** | 批量选中节点，支持批量标记/导出 | G6 框选交互 |
| **子图导出** | 将当前可视区域导出为 PNG/SVG 图片 | 用于嵌入报告 |
| **风险滤镜** | 按风险等级筛选节点 (高/中/低/无) | 颜色 + 筛选联动 |
| **全文搜索** | 搜索框输入实体名称，自动定位 | 模糊匹配 + 自动居中 |

### 4.8.3 画布控制

- 缩放 (0.1x ~ 5x)
- 全图适配 (FitView)
- 撤销/重做 (操作历史栈，深度 20)
- 布局切换 (力导向 / 分层 / 环形 / 网格 / 同心圆)

## 4.9 知识图谱风险问答 (KnowledgeQA)

面向终端用户的智能交互入口。用户通过自然语言提出风险相关问题，后端运行 DRA-MA 多阶段分析管线，前端**实时展示推理过程**、生成关联知识图谱并提供数据分析能力。

### 4.9.1 核心价值

| 维度 | 描述 |
|------|------|
| **交互自然化** | 支持中文自然语言输入，用户无需掌握 Cypher 或图查询语言 |
| **推理透明化** | 实时流式展示管线各阶段执行状态和中间结果 |
| **结果可视化** | 推理结果自动关联知识图谱子图，以 G6 力图展示实体与关系网络 |
| **分析深度化** | 支持路径追踪、异常发现、法规匹配，覆盖股权传导、人事关联、担保链、资金轨迹、事件因果 |

### 4.9.2 前端架构

采用**双 Zustand Store** 模式:

```
┌─────────────────────────────────────┐
│         agentStore (内存)            │
│  当前会话工作状态:                    │
│  - messages[] / currentSubgraph     │
│  - analysisResult / currentRoute    │
│  - riskScores / governancePlan      │
│  - expandedCommunityResult          │
│  - expandedCommunityId              │
│  - selectedRiskPathId               │
│  方法: sendMessage(), clearHistory() │
└──────────────┬──────────────────────┘
               │ auto-save (useEffect debounce 1s)
               ▼
┌─────────────────────────────────────┐
│     chatStore (localStorage 持久化)  │
│  多会话持久化:                       │
│  - sessions[] / activeSessionId     │
│  方法: createNewSession(),          │
│        switchSession(),              │
│        deleteSession()              │
└─────────────────────────────────────┘
```

### 4.9.3 界面布局

三栏式布局: 左侧栏 (会话管理 260px) + 中间栏 (聊天窗口 480px) + 右侧栏 (图谱视图 / 治理报告 / 分析面板 / 群体图谱)

右侧面板通过 Segmented 切换四种视图:
- **图谱视图**: G6 力导向图 + 社区着色 + 图例面板
- **治理报告**: 风险评分详情 + 风险路径 (含群体标签链) + 综合报告 + 协同治理方案
- **分析面板**: ECharts 图表 + 数据预览表格
- **群体图谱**: 两级缩放社区图 (社区聚合 → 成员展开) + 风险路径联动高亮

### 4.9.4 上下文管理

- 双击图谱节点将实体注入聊天上下文 (`[EntityName]` 前缀)
- ContextTagBar 水平标签栏，支持删除单个/清空全部
- 上限 5 个实体标签，超出显示 "+N more"
