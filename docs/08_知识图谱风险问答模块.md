# 08 — 知识图谱风险问答模块

## 8.1 模块概述

BiDA-KG 知识图谱风险问答模块（Knowledge Q&A）是面向终端用户的智能交互入口。用户通过自然语言提出风险相关问题，后端调用 DRA-MA（Dynamic Role Assignment Multi-Agent）多智能体架构进行协同推理，前端**实时展示推理过程**、生成关联知识图谱并提供数据分析能力。

### 8.1.1 核心价值

| 维度 | 描述 |
|------|------|
| **交互自然化** | 支持中文自然语言输入，用户无需掌握 Cypher 或图查询语言 |
| **推理透明化** | 实时流式展示多智能体思考过程（Planner → Retriever → Analyst → Compliance → Reporter） |
| **结果可视化** | 推理结果自动关联知识图谱子图,以 G6 力图展示实体与关系网络 |
| **分析深度化** | 支持路径追踪、异常发现、法规匹配，分析维度覆盖股权传导、人事关联、担保链、资金轨迹、事件因果 |

### 8.1.2 在系统中的位置

```
┌─────────────────────────────────────────────────────────────┐
│                    业务应用层                                │
│  ┌──────────┐  ┌──────────────┐  ┌──────────────────────┐  │
│  │ 风险仪表盘 │  │ 知识图谱检索  │  │ 风险问答 (本模块)    │  │
│  │ Welcome   │  │ GeneralPage  │  │ KnowledgeQA          │  │
│  └──────────┘  └──────────────┘  └──────────────────────┘  │
├─────────────────────────────────────────────────────────────┤

│   Planner → Retriever → Analyst → Compliance → Reporter     │
├─────────────────────────────────────────────────────────────┤
```

## 8.3 DRA-MA 多智能体架构

### 8.3.1 五大智能体角色

本模块后端基于 DRA-MA 架构，包含五个专属智能体：

| 智能体 | 角色标识 | 职责 | 输入 | 输出 |
|--------|---------|------|------|------|
| **Planner** | `planning` | 接收用户问题，拆解为子任务，编排执行计划 | 原始自然语言问题 | 结构化任务计划 (JSON) |
| **Retriever** | `retrieving` | Text-to-Cypher 转换，从 Neo4j 提取相关子图 | 信息需求描述 | 子图数据（节点 + 关系） |
| **Analyst** | `analyzing` | 风险路径推理、异常模式识别、风险量化评分 | 子图数据 + 分析维度 | 风险路径、异常发现、风险评级 |
| **Compliance** | `compliance` | 法规匹配、违规认定、处置建议 | Analyst 结论 + 法规层数据 | 违规认定、法条引用、建议动作 |
| **Reporter** | `reporting` | 汇总所有发现，生成结构化风险报告 | 全部前置分析结果 | Markdown 报告 + 结构化 JSON |

### 8.3.2 推理流程

```
用户问题: "某公司是否存在股权穿透风险"
    │
    ▼
[Planner]  拆解为子任务:
           ├── 查询某公司股权结构 (Retriever)
           ├── 查询关联方信息 (Retriever)
           └── 分析股权穿透风险 (Analyst)
    │
    ▼
[Retriever]  执行 Cypher 查询:
             MATCH (c:COMPANY {name: $name})-[r:INVEST*1..5]-(related)
             RETURN c, r, related
    │
    ▼
[Analyst]  分析子图:
           ├── 识别控制链路径
           ├── 检测循环持股
           └── 评分穿透风险等级
    │
    ▼
[Compliance]  法规匹配:
              ├── 《公司法》关联交易条款
              └── 《证券法》信息披露要求
    │
    ▼
[Reporter]  生成报告:
            ├── 执行摘要
            ├── 风险路径 (含影响权重)
            ├── 异常发现 (含置信度)
            ├── 合规匹配 (含法条引用)
            └── 建议措施
```

### 8.3.3 SSE 事件流规范

后端通过 SSE (Server-Sent Events) 向前端推送推理进度和结果：

| 事件类型 | event 字段 | data 字段 | 触发时机 |
|---------|-----------|-----------|---------|
| 阶段变更 | `stage` | `{"stage": "planning", "content": "正在拆解任务..."}` | 每个 Agent 开始时 |
| 图谱数据 | `graph` | `{"nodes": [...], "edges": [...]}` | Retriever 完成查询后 |
| 推荐卡片 | `cards` | `{"recommendations": [...]}` | Analyst 输出推荐实体时 |
| 分析结果 | `analysis_text` | `{"content": "分析文本片段..."}` | Analyst 流式输出时 (增量) |
| 图表配置 | `echarts_config` | `{...完整 ECharts option...}` | Analyst 输出图表配置时 |
| 原始数据 | `raw_data` | `{"columns": [...], "rows": [[...], ...]}` | Analysis 路由完成时 |
| 综述报告 | `review` | `{"overall": "...", "highlights": [...]}` | Reporter 输出最终报告时 |
| 完成 | `done` | `{"message": "分析完成"}` | 全部流程结束时 |
| 错误 | `error` | `{"error": "错误描述"}` | 任何阶段出错时 |

### 8.3.4 协同治理模块核心接口汇总表

当前代码中，协同治理主流程不是由多个独立 HTTP 接口逐步串联，而是以 `POST /api/v1/chat/unified-stream` 作为统一入口，通过 SSE 事件依次推送意图识别、实体对齐、证据子图、群体发现、风险传导、合规分析和报告生成结果。下表按用户输入后的真实调用链整理。

| API | 输入 | 输出 | 核心用途 |
|-----|------|------|---------|
| 统一分析 API：`POST /api/v1/chat/unified-stream` | JSON：`query`、`sessionId`、`roundId`、`maxHop`、`intentHint`、`fileContent` | SSE 事件流：`stage`、`entities`、`subgraph`、`entity_stats`、`community`、`entity_community_map`、`candidate_risk_paths`、`risk_paths`、`anomaly_findings`、`compliance`、`compliance_scores`、`compliance_indicators`、`scoring`、`governance`、`report`、`done`、`error` | 协同治理主入口。前端普通问答和社区风险报告均优先走该接口；`intentHint=graph_qa` 时展示查询子图和问答回复，`intentHint=risk_analysis` 时进入协同治理报告流程 |
| 意图路由 API：`POST /api/v1/chat/route` | JSON：`query` | JSON：`route` 为 `graph` / `risk` / `clarify`，以及 `clarify_message` | 辅助判断用户问题类型。当前前端也有关键词兜底：普通“有关联/有哪些公司”走图谱问答，明确“风险/合规/传导/社区报告/协同治理”走风险分析 |
| 普通图谱问答 API：`POST /api/v1/chat/recommend` | JSON：`query`、`history`、`sessionId`、`roundId` | JSON：图谱问答结果、推荐实体、子图节点/关系或推荐卡片 | 非流式兼容接口，适合简单关系查询。当前主页面实际更常用 `/chat/unified-stream` 的 `graph_qa` 分支 |
| 实体识别与实体对齐事件：`/chat/unified-stream` → `entities` | 输入来自统一分析 API 的 `query`；内部由 `IntentAgent` 抽取原始实体，并由 `EntityResolver` 对齐 KG 实体 | SSE：`resolved`、`unresolved`、`resolved_count`、`unresolved_count`，实体项包含 `raw`、`canonical_name`、`kg_node_id`、`match_type`、`confidence` | 从用户问题中识别风险主体或查询主体。对“鑫达”等简称，当前前端会先反问补充全称；后端实体对齐负责防止 LLM 直接猜错不存在的实体 |
| 子图查询 API/事件：`/chat/unified-stream` → `subgraph`；交互展开：`GET /api/v1/graph/expand/{node_id}` | `subgraph` 输入：用户问题、已对齐实体、`maxHop`；`expand` 输入：`node_id`、`depth`、`limit` | `subgraph` SSE：`nodes`、`edges`、`node_count`、`edge_count`、`relation_types`；`expand` 返回直接关联子图 `nodes`、`edges` | 查询证据子图。简单问答只展示查询子图谱；图谱中双击节点调用展开接口获取直接子图 |
| 群体发现 API/事件：`/chat/unified-stream` → `community`、`entity_community_map`；局部种子社区：`POST /api/v1/graph/communities/seed-subgraph` | 统一流输入：当前证据子图；种子社区输入：`seedNames`、`seedIds`、`maxHop`、`method`、`minCommunitySize`、`pathLimit` | `community`：群体列表、成员数、算法；`entity_community_map`：实体归属群体和角色；`seed-subgraph`：`seed_nodes`、`subgraph`、`connected_subgraph`、`communities`、`entity_community_map`、`visualization.flow` | 发现风险主体所在群体、最大连通子图和社区结构，为“风险主体 → 群体发现 → 风险传导路径 → 协同治理社区报告”提供社区视图 |
| 风险传导 API/事件：`/chat/unified-stream` → `candidate_risk_paths`、`risk_paths`；兼容流：`GET /api/v1/risk/analyze-stream` | 输入：`query`、`sessionId`、`roundId`、`maxHop`、可选 `communityId`、`focusEntities`、`fileContent` | 风险路径：`path_id`、`risk_level`、`affected_entities`、`node_ids`、`edge_ids`、`path_description`、`confidence`；并同步写入子图 `paths` 用于高亮 | 识别主体之间、群体之间的风险传导路径。前端支持“群体间传导”和“具体节点路径”两个视图 |
| 合规分析 API/事件：`/chat/unified-stream` → `compliance`、`compliance_scores`、`compliance_indicators` | 输入：风险路径、异常发现、证据子图、法规/合规匹配上下文 | `compliance`：法规、条款、违规点、建议动作、置信度；`compliance_scores`：节点/指标得分；`compliance_indicators`：一级/二级/三级合规指标、客观分和证据 | 对风险行为进行法规匹配和合规评分，驱动前端“合规分析”面板 |
| 协同治理社区报告 API/事件：`/chat/unified-stream` → `governance`、`report`；兼容接口：`POST /api/v1/risk/analyze` | 输入：实体统计、社区发现、实体-社区映射、风险路径、异常发现、合规匹配、风险评分 | `governance`：治理动作、责任部门、优先级、升级规则；`report`：`executive_summary`、`risk_paths`、`anomaly_findings`、`compliance_matches`、`governance_plan`、`markdown_report`、`recommendations` | 汇总上游结果，生成“协同治理社区报告”，并切换前端到治理报告面板 |
| 报告历史 API：`GET /api/v1/risk/reports`、`GET /api/v1/risk/reports/{report_id}` | 查询参数：`page`、`limit`、`risk_level`；路径参数：`report_id` | 报告列表或报告详情，包括报告编号、时间、风险等级、摘要、Markdown 内容等 | 查看历史协同治理报告 |
| 报告导出 API：`POST /api/v1/risk/reports/export-docx` | JSON：`report`、`reportId`、`queryText` | `.docx` 文件二进制流 | 将当前协同治理报告导出为 Word 文档 |

### 8.4.3 状态管理架构

采用 **双 Zustand Store** 模式，职责分离：

```
┌─────────────────────────────────────┐
│         agentStore (内存)            │
│  ┌─────────────────────────────┐    │
│  │ 当前会话工作状态               │    │
│  │  - messages[]                │    │
│  │  - currentSubgraph           │    │
│  │  - analysisResult            │    │
│  │  - currentRoute              │    │
│  │  - isLoading / error         │    │
│  │  - pendingRecommendations    │    │
│  │  - clarifyMessage            │    │
│  └─────────────────────────────┘    │
│  方法: sendMessage(), clearHistory() │
└──────────────┬──────────────────────┘
               │ auto-save (useEffect debounce 1s)
               ▼
┌─────────────────────────────────────┐
│     chatStore (localStorage 持久化)  │
│  ┌─────────────────────────────┐    │
│  │ 多会话持久化                  │    │
│  │  - sessions[]               │    │
│  │  - activeSessionId          │    │
│  └─────────────────────────────┘    │
│  方法: createNewSession(),           │
│        switchSession(),              │
│        deleteSession(),              │
│        renameSession(),              │
│        updateCurrentSession()        │
└─────────────────────────────────────┘
```

**数据流：**
1. 用户发送消息 → `agentStore.sendMessage()` → SSE 流式更新 `messages`/`currentSubgraph`/`analysisResult`
2. `index.tsx` useEffect 监听变化 → 1 秒防抖后调用 `chatStore.updateCurrentSession()` 持久化
3. 用户切换会话 → `chatStore.switchSession()` → 从 `sessions[]` 恢复 `agentStore` 状态

## 8.5 API 接口规范

### 8.5.1 接口总览

| 接口路径 | 方法 | 用途 | 响应类型 |
|---------|------|------|---------|
| `/api/v1/chat/route` | POST | 意图路由分发 | JSON |
| `/api/v1/chat/recommend-stream` | GET | 图谱推荐 (SSE 流式) | SSE EventStream |
| `/api/v1/chat/analyze` | POST | 数据分析 (SSE 流式) | SSE EventStream |
| `/api/v1/graph/expand/{node_id}` | GET | 图谱节点展开 | JSON |
| `/health` | GET | API 健康检查 | JSON |

### 8.5.2 意图路由接口

**请求：**
```json
POST /api/v1/chat/route
Content-Type: application/json

{
  "query": "某公司是否存在股权穿透风险",
  "history": [
    {"role": "user", "content": "..."},
    {"role": "assistant", "content": "..."}
  ]
}
```

**响应：**
```json
{
  "route": "graph",
  "clarify_message": null
}
```

**路由规则：**

| route 值 | 触发条件 | 后续动作 |
|---------|---------|---------|
| `"graph"` | 涉及实体查询、关系查询、风险路径 | 调用 `/chat/recommend-stream` (SSE) |
| `"analysis"` | 涉及统计、聚合、趋势分析 | 调用 `/chat/analyze` (SSE) |
| `"clarify"` | 问题模糊、缺少关键实体 | 展示 `clarify_message` 提示用户细化 |

### 8.5.3 图谱推荐流式接口

**请求：**
```
GET /api/v1/chat/recommend-stream?query=某公司股权穿透&focus_entities=["某公司"]&max_hop=3
```

**SSE 事件序列示例：**

```
event: stage
data: {"stage": "planning", "content": "正在拆解分析任务..."}

event: stage
data: {"stage": "retrieving", "content": "正在从知识图谱检索相关实体..."}

event: cards
data: {"recommendations": [{"itemId": "...", "title": "某公司", "score": 0.95, "riskLevel": -2, "type": "COMPANY", "highlight": "股权穿透风险高"}]}

event: graph
data: {"nodes": [{"id": "...", "type": "COMPANY", "properties": {...}}], "edges": [{"source": "...", "target": "...", "relation": "INVEST"}]}

event: stage
```

## 8.10 会话管理详细设计

### 8.10.1 会话数据结构

```typescript
interface Session {
  id: string;           // crypto.randomUUID()
```

### 8.10.2 会话操作

|------|--------|------|

- **恢复时机**：页面加载时自动从 localStorage 恢复
- **存储上限**：受浏览器限制 (~5MB)，超过时提示用户清理旧会话

## 8.11 聊天消息与流式输出

### 8.11.1 消息类型

| 角色 | role 值 | 渲染方式 | 颜色 |
|------|--------|---------|------|
| 用户 | `user` | 右对齐蓝色气泡 | `#1890ff` 背景 |
| 助手 | `assistant` | 左对齐白色气泡，支持 Markdown | `#fff` 背景 |
| 系统 | `system` | 居中灰色提示条 | `#f5f5f5` 背景 |

### 8.11.2 流式输出行为

1. 用户点击发送 → 立即在消息列表追加用户消息和占位 assistant 消息 (`isLoading: true`)
2. SSE 接收到 `stage` 事件 → 更新助理消息的 `thinkingStatus`，显示为脉冲动画文本
3. SSE 接收到 `cards` 事件 → 在助理消息下方插入 `RiskEntityCard` 列表
4. SSE 接收到 `review` 事件 → 更新助理消息 `content`，`isLoading` 切换为 `false`
5. SSE 接收到 `graph` 事件 → 切换到图谱面板，渲染 G6 力图
6. SSE 接收到 `error` 事件 → 助理消息显示错误信息 + "Retry" 按钮
7. SSE 接收到 `done` 事件 → 完成加载状态，触发自动保存

### 8.11.3 输入框快捷键

| 快捷键 | 行为 |
|--------|------|
| `Enter` | 发送消息 |
| `Shift + Enter` | 插入换行符 |
| `Escape` | 清空输入框 |

## 8.15 性能需求

| 指标 | 目标值 | 测量方式 |
|------|--------|---------|
| 首屏加载 (FCP) | < 2s | Lighthouse |
| 消息发送到首 token 显示 | < 3s | SSE `stage` 事件时间戳 |
| G6 图谱渲染 (1000 节点) | < 2s | `graph.render()` 前后 time delta |
| ECharts 图表渲染 | < 1s | `setOption()` 前后 time delta |
| Session 切换恢复 | < 500ms | `switchSession()` → 界面更新完成 |
| SSE 流式接收延迟 | < 200ms per event | EventSource onmessage 时间戳间隔 |

## 8.16 安全性

| 安全项 | 措施 |
|--------|------|
| API 认证 | JWT Token (继承系统统一认证) |
| Cypher 注入防护 | 后端参数化查询 + 禁止 CREATE/DELETE/SET/MERGE/DROP |
| XSS 防护 | React 默认转义 + Markdown 渲染使用安全解析器 |
| 敏感数据遮蔽 | LLM 调用前自动遮蔽 PII (身份证号、手机号等) |
| 会话隔离 | 每个 session 独立 UUID，客户端 localStorage 隔离 |
| 审计日志 | 记录所有 API 调用 (query, timestamp, sessionId, userId) |
