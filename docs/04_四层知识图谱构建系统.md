# 04 — 四层知识图谱构建系统

## 4.1 模块概述

将清洗后的数据转化为结构化的知识网络，基于 Neo4j 图数据库存储。这是整个系统的知识底座，连接"数据"与"智能"。

---

## 4.2 本体模型设计 (Ontology)

### 4.2.1 整体结构

```
┌──────────────────────────────────────────────┐
│  法规/行动层 (Regulation & Action Layer)      │
│  Node: Regulation, Policy, Action             │
│  Rel:  REGULATES, PRESCRIBES, MITIGATES       │
│  例：《公司法》第20条 → 规制 → 抽逃出资行为      │
└──────────────────┬───────────────────────────┘
                   │ REFINES / CONSTRAINS
┌──────────────────┴───────────────────────────┐
│  特征层 (RiskFeature Layer)                   │
│  Node: RiskFeature, RiskPattern               │
│  Rel:  EXHIBITS, TRIGGERS                     │
│  例：公司A → 表现 → "资金链断裂特征"            │
└──────────────────┬───────────────────────────┘
                   │ DETECTED_FROM
┌──────────────────┴───────────────────────────┐
│  事件层 (Event Layer)                         │
│  Node: Investment, Litigation, Penalty, News   │
│  Rel:  INVESTS_IN, SUES, SANCTIONS, REPORTS   │
│  例：公司A → 被列为被执行人 → 执行案件001        │
└──────────────────┬───────────────────────────┘
                   │ INVOLVES
┌──────────────────┴───────────────────────────┐
│  主体层 (Subject Layer)                       │
│  Node: Company, Person, Institution, Product   │
│  Rel:  CONTROLS, EMPLOYS, GUARANTEES, SUPPLIES│
│  例：张三 → 担任 → 法人代表 → 公司A             │
└──────────────────────────────────────────────┘
```

### 4.2.2 主体层 (Subject Layer)

定义系统中最基础的实体类型及其属性。

#### Company 实体

| 属性 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `uscid` | String | Y | 统一社会信用代码 (唯一标识) |
| `name` | String | Y | 企业全称 |
| `short_name` | String | N | 企业简称 |
| `legal_person` | String | N | 法定代表人 |
| `registered_capital` | Float | N | 注册资本 (万元) |
| `established_date` | Date | N | 成立日期 |
| `status` | Enum | Y | 经营状态：存续/注销/吊销/迁出 |
| `industry` | String | N | 所属行业 (国民经济行业分类) |
| `province` / `city` | String | N | 注册地 |

#### Person 实体

| 属性 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | String | Y | 姓名 |
| `id_type` | Enum | N | 证件类型 |
| `id_number_hash` | String | N | 证件号哈希（脱敏存储） |
| `gender` | Enum | N | 性别 |

#### Institution 实体

| 属性 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | String | Y | 机构全称 |
| `type` | Enum | Y | 类型：政府/司法/金融机构/交易所 |

### 4.2.3 事件层 (Event Layer)

定义实体的动态行为。

| 事件类型 | 关键属性 | 关联主体 |
|---------|---------|---------|
| **Investment** (投资) | 投资金额、比例、时间 | (Company)-[:INVESTS_IN]->(Company) |
| **Employment** (任职) | 职位、入职时间、离职时间 | (Person)-[:EMPLOYED_BY]->(Company) |
| **Litigation** (诉讼) | 案由、诉讼地位、案号、日期 | (Company/Person)-[:INVOLVED_IN]->(Litigation) |
| **Penalty** (处罚) | 处罚类型、原因、金额、机关 | (Company/Person)-[:RECEIVES]->(Penalty) |
| **Guarantee** (担保) | 担保金额、担保类型、起始日期 | (Company)-[:GUARANTEES]->(Company) |
| **FundFlow** (资金流动) | 金额、方向、时间 | (Entity)-[:TRANSFERS_TO]->(Entity) |
| **NewsEvent** (新闻) | 标题、摘要、情感、来源 URL | (Entity)-[:MENTIONED_IN]->(NewsEvent) |

### 4.2.4 特征层 (RiskFeature Layer)

抽象出的风险标签，由规则引擎或 Agent 推理自动挂载。

| 特征类型 | 识别方式 | 示例 |
|---------|---------|------|
| **资金链断裂特征** | 规则：被执行金额 > 净资产 * 80% | "高负债风险" |
| **高管频繁变更** | 规则：12 个月内法定代表人变更 ≥ 2 次 | "治理不稳定风险" |
| **关联交易密集** | 算法：关联交易占比 > 营收 50% | "利益输送嫌疑" |
| **隐性关联** | Graph 算法：共同高管 / 共同股东 / 同址 | "隐性一致行动人" |
| **空壳公司** | 规则：无实际经营地 + 无社保记录 + 认缴制未实缴 | "壳公司风险" |
| **异常资金环流** | 算法：资金 A→B→C→A 闭环检测 | "资金循环流转嫌疑" |

### 4.2.5 法规/行动层 (Action & Regulation Layer)

| 节点类型 | 关键属性 | 示例 |
|---------|---------|------|
| **Regulation** | 法规名称、文号、颁布机关、生效日期、条款内容 | 《公司法》第 20 条：禁止滥用公司法人独立地位 |
| **Policy** | 政策名称、发布机构、发布日 | 《关于加强金融风险防控的指导意见》 |
| **Action** | 处置类型、依据法条、执行部门 | 冻结账户、限制高消费、发函问询 |

## 4.3 知识抽取

### 4.3.1 抽取流水线

```
非结构化文本 → 分句/分段 → NER 实体识别 → RE 关系抽取 → 实体链接 → 写入图谱
```

### 4.3.2 双引擎策略

| 引擎 | 适用场景 | 优势 | 劣势 |
|------|---------|------|------|
| **规则引擎** (spaCy + 词表匹配) | 结构化程度高的文本（裁判文书、工商变更记录） | 速度快、可解释、成本低 | 泛化能力弱 |
| **LLM 引擎** (DeepSeek / GPT) | 非结构化文本（新闻、社交媒体、研报） | 语义理解强、泛化好 | 速度慢、成本高、需 Prompt 调优 |

**切换规则：** 默认使用规则引擎，当规则引擎置信度 < 阈值时，fallback 至 LLM 引擎。系统通过 `NER_MODEL` 环境变量控制：`rule` / `llm` / `hybrid`。

## 4.4 知识融合与对齐

### 4.4.1 实体对齐

将多源数据中指向同一现实实体的节点进行合并：

| 对齐策略 | 适用场景 | 方法 |
|---------|---------|------|
| **精确匹配** | 统一社会信用代码、身份证哈希 | 唯一键匹配 |
| **模糊匹配** | 企业名称（含曾用名、简称） | TfIdf + 余弦相似度，阈值 ≥ 0.85 |
| **图结构匹配** | 无名实体或信息残缺 | 邻居节点相似度 + 图嵌入 |

### 4.4.2 冲突解决

多源数据对同一属性有不同值时：
- **时间戳优先：** 取最新数据
- **权威度优先：** 官方数据源 > 第三方 > 网络采集
- **置信度标记：** 冲突无法解决时，保留多值并标记 `confidence: 0.5`

## 4.5 图谱存储与查询服务

### 4.5.1 Neo4j 存储规范

| 配置项 | 建议值 | 说明 |
|--------|--------|------|
| 节点标签 | 按四层分类 (Subject, Event, Feature, Regulation) | 便于按层查询 |
| 关系类型 | 大写 + 下划线，语义化命名 | 如 `CONTROLS`, `INVESTS_IN` |
| 索引 | 唯一标识字段建索引 | `uscid`, `name`, `id_hash` |
| 分区 | 按行业/地域分区 | 大规模部署时 |

### 4.5.2 对外查询 API

| 接口 | 描述 | 响应时间要求 |
|------|------|------------|
| `/kg/entity/{id}` | 单实体详情 (含 1 跳邻居) | < 200ms |
| `/kg/expand` | 从指定节点 N 跳扩展子图 | < 500ms (3 跳) |
| `/kg/path` | 两节点间最短路径 / 所有路径 | < 1s (5 跳) |
| `/kg/community` | 社区发现 (Louvain / Label Propagation) | < 5s |
| `/kg/search` | 全文检索实体 | < 300ms |
| `/kg/cypher` | 受控 Cypher 直查 (只读) | 取决于查询复杂度 |
