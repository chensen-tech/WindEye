# 04 — 四层知识图谱构建系统

## 4.1 模块概述

将清洗后的数据转化为结构化的知识网络，基于 Neo4j 图数据库存储。这是整个系统的知识底座，连接"数据"与"智能"。

---

## 4.2 本体模型设计 (Ontology)

### 4.2.1 整体结构

```
┌──────────────────────────────────────────────┐
│  法规/行动层 (Regulation & Action Layer)      │
│  Node: Regulation, Policy, Action             │
│  Rel:  REGULATES, PRESCRIBES, MITIGATES       │
│  例：《公司法》第20条 → 规制 → 抽逃出资行为      │
└──────────────────┬───────────────────────────┘
                   │ REFINES / CONSTRAINS
┌──────────────────┴───────────────────────────┐
│  特征层 (RiskFeature Layer)                   │
│  Node: RiskFeature, RiskPattern               │
│  Rel:  EXHIBITS, TRIGGERS                     │
│  例：公司A → 表现 → "资金链断裂特征"            │
└──────────────────┬───────────────────────────┘
                   │ DETECTED_FROM
┌──────────────────┴───────────────────────────┐
│  事件层 (Event Layer)                         │
│  Node: Investment, Litigation, Penalty, News   │
│  Rel:  INVESTS_IN, SUES, SANCTIONS, REPORTS   │
│  例：公司A → 被列为被执行人 → 执行案件001        │
└──────────────────┬───────────────────────────┘
                   │ INVOLVES
┌──────────────────┴───────────────────────────┐
│  主体层 (Subject Layer)                       │
│  Node: Company, Person, Institution, Product   │
│  Rel:  CONTROLS, EMPLOYS, GUARANTEES, SUPPLIES│
│  例：张三 → 担任 → 法人代表 → 公司A             │
└──────────────────────────────────────────────┘
```

### 4.2.2 主体层 (Subject Layer)

定义系统中最基础的实体类型及其属性。

#### Company 实体

| 属性 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `uscid` | String | Y | 统一社会信用代码 (唯一标识) |
| `name` | String | Y | 企业全称 |
| `short_name` | String | N | 企业简称 |
| `legal_person` | String | N | 法定代表人 |
| `registered_capital` | Float | N | 注册资本 (万元) |
| `established_date` | Date | N | 成立日期 |
| `status` | Enum | Y | 经营状态：存续/注销/吊销/迁出 |
| `industry` | String | N | 所属行业 (国民经济行业分类) |
| `province` / `city` | String | N | 注册地 |

#### Person 实体

| 属性 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | String | Y | 姓名 |
| `id_type` | Enum | N | 证件类型 |
| `id_number_hash` | String | N | 证件号哈希（脱敏存储） |
| `gender` | Enum | N | 性别 |

#### Institution 实体

| 属性 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | String | Y | 机构全称 |
| `type` | Enum | Y | 类型：政府/司法/金融机构/交易所 |

### 4.2.3 事件层 (Event Layer)

定义实体的动态行为。

| 事件类型 | 关键属性 | 关联主体 |
|---------|---------|---------|
| **Investment** (投资) | 投资金额、比例、时间 | (Company)-[:INVESTS_IN]->(Company) |
| **Employment** (任职) | 职位、入职时间、离职时间 | (Person)-[:EMPLOYED_BY]->(Company) |
| **Litigation** (诉讼) | 案由、诉讼地位、案号、日期 | (Company/Person)-[:INVOLVED_IN]->(Litigation) |
| **Penalty** (处罚) | 处罚类型、原因、金额、机关 | (Company/Person)-[:RECEIVES]->(Penalty) |
| **Guarantee** (担保) | 担保金额、担保类型、起始日期 | (Company)-[:GUARANTEES]->(Company) |
| **FundFlow** (资金流动) | 金额、方向、时间 | (Entity)-[:TRANSFERS_TO]->(Entity) |
| **NewsEvent** (新闻) | 标题、摘要、情感、来源 URL | (Entity)-[:MENTIONED_IN]->(NewsEvent) |

### 4.2.4 特征层 (RiskFeature Layer)

抽象出的风险标签，由规则引擎或 Agent 推理自动挂载。

| 特征类型 | 识别方式 | 示例 |
|---------|---------|------|
| **资金链断裂特征** | 规则：被执行金额 > 净资产 * 80% | "高负债风险" |
| **高管频繁变更** | 规则：12 个月内法定代表人变更 ≥ 2 次 | "治理不稳定风险" |
| **关联交易密集** | 算法：关联交易占比 > 营收 50% | "利益输送嫌疑" |
| **隐性关联** | Graph 算法：共同高管 / 共同股东 / 同址 | "隐性一致行动人" |
| **空壳公司** | 规则：无实际经营地 + 无社保记录 + 认缴制未实缴 | "壳公司风险" |
| **异常资金环流** | 算法：资金 A→B→C→A 闭环检测 | "资金循环流转嫌疑" |

### 4.2.5 法规/行动层 (Action & Regulation Layer)

| 节点类型 | 关键属性 | 示例 |
|---------|---------|------|
| **Regulation** | 法规名称、文号、颁布机关、生效日期、条款内容 | 《公司法》第 20 条：禁止滥用公司法人独立地位 |
| **Policy** | 政策名称、发布机构、发布日 | 《关于加强金融风险防控的指导意见》 |
| **Action** | 处置类型、依据法条、执行部门 | 冻结账户、限制高消费、发函问询 |

## 4.3 知识抽取

### 4.3.1 抽取流水线

```
非结构化文本 → 分句/分段 → NER 实体识别 → RE 关系抽取 → 实体链接 → 写入图谱
```

### 4.3.2 双引擎策略

| 引擎 | 适用场景 | 优势 | 劣势 |
|------|---------|------|------|
| **规则引擎** (spaCy + 词表匹配) | 结构化程度高的文本（裁判文书、工商变更记录） | 速度快、可解释、成本低 | 泛化能力弱 |
| **LLM 引擎** (DeepSeek / GPT) | 非结构化文本（新闻、社交媒体、研报） | 语义理解强、泛化好 | 速度慢、成本高、需 Prompt 调优 |

**切换规则：** 默认使用规则引擎，当规则引擎置信度 < 阈值时，fallback 至 LLM 引擎。系统通过 `NER_MODEL` 环境变量控制：`rule` / `llm` / `hybrid`。

## 4.4 知识融合与对齐

### 4.4.1 实体对齐

将多源数据中指向同一现实实体的节点进行合并：

| 对齐策略 | 适用场景 | 方法 |
|---------|---------|------|
| **精确匹配** | 统一社会信用代码、身份证哈希 | 唯一键匹配 |
| **模糊匹配** | 企业名称（含曾用名、简称） | TfIdf + 余弦相似度，阈值 ≥ 0.85 |
| **图结构匹配** | 无名实体或信息残缺 | 邻居节点相似度 + 图嵌入 |

### 4.4.2 冲突解决

多源数据对同一属性有不同值时：
- **时间戳优先：** 取最新数据
- **权威度优先：** 官方数据源 > 第三方 > 网络采集
- **置信度标记：** 冲突无法解决时，保留多值并标记 `confidence: 0.5`

## 4.5 事理知识图谱构建核心 API

事理知识图谱构建主链路是：数据采集 → ETL 管线 → Dify 提取 → Neo4j 入库。当前后端接口主要集中在 `backend/api/pipeline_routes.py`，统一挂载在 `/api/v1/pipeline` 下；文件文本抽取接口位于 `backend/api/router.py` 的 `/api/v1/chat/upload`。当前代码没有实现 `/api/v1/pipeline/files/{source}/upload`，构建源文件通过磁盘数据源目录扫描进入管线。

| API | 输入 | 输出 | 核心用途 |
|-----|------|------|---------|
| `GET /api/v1/pipeline/status` | 无 | `status=idle/running`；运行中返回 `current_run`，包含 `source`、起止阶段、启动时间和状态 | 查询当前知识图谱构建任务是否正在运行 |
| `POST /api/v1/pipeline/run` | Query：`source` 必填；可选 `start_stage`、`end_stage` | 触发结果：`message`、`source`、`start_stage`、`end_stage` | 启动 ETL 主流程，按 `crawl → parse → extract → link → resolve → import → index` 执行 |
| `GET /api/v1/pipeline/sources` | 无 | `sources` 列表、`count`；每个数据源含 `key`、`name`、`category`、`layer`、`entity_types`、`relation_types` | 获取配置化数据源，用于选择要构建的图谱来源 |
| `GET /api/v1/pipeline/runs` | Query：`limit`、`include_data` | `runs` 历史记录、`total`；可选包含 `_records`、`_entities` 明细 | 查看历史 ETL 构建记录和各阶段统计 |
| `GET /api/v1/pipeline/entities/{source}` | Path：`source` 数据源标识 | `success`、`data`；按阶段返回 `records_processed`、`stats`、可选 `entities`、`records` | 查看某个数据源最近一次构建中解析和抽取出的实体数据 |
| `GET /api/v1/pipeline/data-sources` | 无 | `sources`、`total`；每项包含 `input_glob`、`file_count`、层级和实体/关系类型 | 扫描磁盘文件数据源，查看每个来源目录中待处理文件数量 |
| `GET /api/v1/pipeline/files/{source}` | Path：`source` 数据源标识 | `source`、`name`、`category`、`layer`、`input_glob`、`files`、`total` | 查看指定数据源目录下的本地待处理文件 |
| `DELETE /api/v1/pipeline/files/{source}` | Path：`source` 数据源标识 | `source`、`deleted`、`message` | 清理指定数据源目录中的文件，避免重复入库 |
| `POST /api/v1/pipeline/crawl/run` | JSON：`mode`、`data_type`、`sources`、`date_start`、`date_end`、`keywords`、`max_pages`、`max_files`、`natural_language_query`、`template_id` | SSE：`start`、`stage`、`source_result`、`complete`、`error` 等事件 | 启动外部数据采集任务，流式返回采集进度，并可触发后续 ETL |
| `GET /api/v1/pipeline/crawl/templates` | 无 | `templates`、`total` | 获取一键采集模板，用于快速发起公告、舆情、风险事件等采集 |
| `POST /api/v1/pipeline/crawl/parse-nl` | Query：`query` 自然语言采集需求 | `success`、`data` 结构化采集参数 | 将自然语言采集需求解析为数据类型、关键词、时间范围、来源等参数 |
| `GET /api/v1/pipeline/crawl/sources` | 无 | `sources`，包含各采集源支持的关键词、时间范围、最大页数等能力 | 查询采集源能力，辅助前端展示可选来源和约束 |
| `GET /api/v1/pipeline/crawl/tasks` | Query：`limit` | `tasks`、`total` | 查看采集任务历史，包括采集模式、数据类型和完成时间 |
| `POST /api/v1/pipeline/extract/{stage}` | Path：`stage=subject_extraction/event_extraction/feature_extraction/regulation_linking`；Query：`source` | `success`、`stage`、`source`、`nodes`、`edges`、`node_count`、`edge_count`、`cypher_statements`、`cypher_preview` | 单阶段调用 Dify 抽取主体、事件、特征或法规映射，返回预览图谱，不直接写入 Neo4j |
| `POST /api/v1/pipeline/dify/import` | Query：`source`、可选 `file_path`、`dry_run` | `success`、`nodes`、`edges`；`dry_run=true` 返回 `cypher_preview`，否则返回 `cypher_executed`、`cypher_errors` | 对法规等文档执行 Dify 抽取，并将生成的 Cypher 直接导入 Neo4j |
| `POST /api/v1/chat/upload` | FormData：`file`，支持 `.txt`、`.md`、`.docx`、`.pdf` | `success`、`data.filename`、`data.text`、`data.char_count`、`data.truncated` | 上传文本类文件并抽取正文，主要供问答/分析使用；当前不是 pipeline 数据源上传接口 |

## 4.6 图谱存储与查询服务

### 4.6.1 Neo4j 存储规范

| 配置项 | 建议值 | 说明 |
|--------|--------|------|
| 节点标签 | 按四层分类 (Subject, Event, Feature, Regulation) | 便于按层查询 |
| 关系类型 | 大写 + 下划线，语义化命名 | 如 `CONTROLS`, `INVESTS_IN` |
| 索引 | 唯一标识字段建索引 | `uscid`, `name`, `id_hash` |
| 分区 | 按行业/地域分区 | 大规模部署时 |

### 4.6.2 对外查询 API

| 接口 | 描述 | 响应时间要求 |
|------|------|------------|
| `/kg/entity/{id}` | 单实体详情 (含 1 跳邻居) | < 200ms |
| `/kg/expand` | 从指定节点 N 跳扩展子图 | < 500ms (3 跳) |
| `/kg/path` | 两节点间最短路径 / 所有路径 | < 1s (5 跳) |
| `/kg/community` | 社区发现 (Louvain / Label Propagation) | < 5s |
| `/kg/search` | 全文检索实体 | < 300ms |
| `/kg/cypher` | 受控 Cypher 直查 (只读) | 取决于查询复杂度 |
